# ML Tag Suggestions

Automatic theme tag suggestions generated by an ONNX image tagger during upload.
Suggestions require human review before being applied. Disabled by default.

---

## Overview

When `ML_TAG_SUGGESTIONS_ENABLED=true`, the arq worker runs ONNX inference on
each uploaded image and stores pending suggestions in `ml_tag_suggestions`. The
uploader (or a moderator with `IMAGE_TAG_ADD`) reviews them via the API;
approvals create `tag_links`, rejections are recorded but do not apply the tag.

**v1 scope: theme tags only.** The pipeline maps WD-Tagger Danbooru vocabulary
to internal theme tags (type=1) via the `tag_mappings` table. Character tag
support is a documented extension point ŌĆö add `CHARACTER_CATEGORY` to the
category filter in `app/services/ml_service.py` and populate character rows in
`tag_mappings`.

**Separate from the report-based suggestion system.** `MlTagSuggestions` is a
distinct model from `ImageReportTagSuggestions`, which stores human suggestions
filed via image reports.

## Setup

### 1. Enable the flag

Add to your `.env`:

```
ML_TAG_SUGGESTIONS_ENABLED=true
```

The worker **fails to start** if this flag is on but model files are missing ŌĆö
intentional, no silent fallback.

### 2. Download the model

```bash
cd ml_models/wd-swinv2-tagger-v3
wget https://huggingface.co/SmilingWolf/wd-swinv2-tagger-v3/resolve/main/model.onnx
```

Or via the HuggingFace CLI:

```bash
huggingface-cli download SmilingWolf/wd-swinv2-tagger-v3 model.onnx \
  --local-dir ml_models/wd-swinv2-tagger-v3
```

See `ml_models/wd-swinv2-tagger-v3/README.md` for model details (~467 MB,
448├Ś448 input, sigmoid already applied in the ONNX graph).

### 3. Optional env vars

| Variable | Default | Description |
|---|---|---|
| `ML_TAG_SUGGESTIONS_ENABLED` | `false` | Master switch |
| `ML_MODELS_PATH` | `ml_models` | Directory holding model subdirectories |
| `ML_MODEL_NAME` | `wd-swinv2-tagger-v3` | Model subdirectory to load |
| `ML_MIN_CONFIDENCE` | `0.35` | Minimum probability threshold |

`ML_MIN_CONFIDENCE` operates on **true probabilities** (sigmoid already applied
in the ONNX graph). The default of 0.35 is calibrated against the fixed model
pipeline; the old branch's 0.6 was calibrated against a now-fixed double-sigmoid
bug.

### 4. Import tag mappings

Populate `tag_mappings` from the curated CSV before enabling suggestions.
Run the migration first to create the `tag_mappings` table, then import:

```bash
uv run alembic upgrade head
uv run python scripts/import_tag_mappings.py
```

This reads `data/tag_mappings.csv` (297 entries: 257 `map` + 40 `ignore`
entries). The script is idempotent ŌĆö skips rows that already exist.

To regenerate a draft mapping CSV against the live database:

```bash
uv run python scripts/generate_tag_mappings.py
# Review data/tag_mappings_draft.csv, then:
uv run python scripts/import_tag_mappings.py data/tag_mappings_draft.csv
```

## How it works

1. **Upload**: `POST /api/v1/images` enqueues `generate_ml_tag_suggestions` with
   a 30-second defer (so thumbnail processing finishes first). Enqueue failure is
   logged but never fails the upload.

2. **Inference**: The arq worker runs ONNX inference, filters predictions below
   `ML_MIN_CONFIDENCE`, resolves Danbooru tags through `tag_mappings` to internal
   tag IDs, skips tags already on the image, and stores `pending` rows in
   `ml_tag_suggestions`.

3. **Review**: The uploader or a moderator reviews suggestions via the API.
   Approvals create `tag_links` (the tag is applied). Rejections are recorded
   without applying the tag.

---

## API Endpoints

All endpoints are under `/api/v1/images/{image_id}/`.

**Permissions**: image uploader or users with `IMAGE_TAG_ADD` permission.

### GET `/api/v1/images/{image_id}/ml-tag-suggestions`

List suggestions. Optional `?status=pending|approved|rejected` filter.
Response includes counts for all three statuses regardless of filter.

```bash
curl -H "Authorization: Bearer $TOKEN" \
  "http://localhost:8000/api/v1/images/12345/ml-tag-suggestions?status=pending"
```

### POST `/api/v1/images/{image_id}/ml-tag-suggestions/review`

Batch approve or reject suggestions. Approving creates a `tag_links` row if the
tag is not already on the image. Idempotent ŌĆö re-reviewing is safe.

```json
POST /api/v1/images/12345/ml-tag-suggestions/review
{
  "suggestions": [
    {"suggestion_id": 1, "action": "approve"},
    {"suggestion_id": 2, "action": "reject"}
  ]
}
```

Response: `{"approved": 1, "rejected": 1, "errors": []}`

### POST `/api/v1/images/{image_id}/ml-tag-suggestions/generate`

Trigger (re-)generation manually.

- Default (`sync=false`): queues background job, returns 202 with `job_id`.
- `?sync=true`: runs inference inline (~0.6s), returns 200 with
  `suggestions_created` count. Useful for testing; loads the ONNX model in the
  API process on first call.
- Returns 503 when `ML_TAG_SUGGESTIONS_ENABLED=false`.

---

## Worker behavior

- Model is loaded once at worker startup when `ML_TAG_SUGGESTIONS_ENABLED=true`.
- Missing model files ŌåÆ worker **fails to start** (no silent degradation).
- Meilisearch unavailability is handled gracefully (warning, not crash).
- On shutdown, `MLTagSuggestionService.cleanup()` is called to release the ONNX
  session.

## Bulk backfill (existing images)

Generating suggestions for the whole corpus is a separate offline job, split
into three stages so GPU inference can run on another host **without database
access**. The reusable logic lives in `app/services/ml_backfill.py`; the stages
are thin CLIs under `scripts/`.

To seed a **production** database from an already-completed run (e.g. a dev box
that is a prod-backup restore ŌĆö the results file is keyed by `image_id` and
stores external tags, so it ports across DBs), see the
[production seeding runbook](ml-tag-suggestions-prod-seeding.md).

CPU throughput is on the order of ~1.7 images/sec on a 10-core box (Ōēł a week for
~1.1M images), so a GPU host is worth it. Inference auto-selects CUDA/ROCm when
the installed onnxruntime build exposes one (`app/services/onnx_providers.py`) ŌĆö
no code change between CPU and GPU runs.

### Stage 1 ŌĆö manifest (needs the DB)

```bash
uv run python scripts/ml_backfill_manifest.py --out manifest.jsonl \
    --missing-theme --exclude-existing
```

Writes `{image_id, filename, ext}` rows. Flags:

- `--missing-theme` ŌĆö only images with no theme tags yet (skips already-themed images)
- `--exclude-existing` ŌĆö skip images that already have ML suggestion rows
- `--status N` / `--all-statuses` ŌĆö status filter (default: active)
- `--limit N` ŌĆö cap the count (smoke tests)

### Stage 2 ŌĆö inference (GPU host; no DB)

Copy the manifest to the inference host and point it at the image store
(`STORAGE_PATH`). Reads thumbnails by default ŌĆö they downscale to the model's
448px input with no quality loss for theme tags, and keep I/O tiny.

```bash
uv run python scripts/ml_backfill_infer.py \
    --manifest manifest.jsonl --out results.jsonl --variant thumbs
```

Sharded and resumable. Split across hosts/processes by shard:

```bash
# host A
uv run python scripts/ml_backfill_infer.py --manifest manifest.jsonl \
    --out r0.jsonl --shards 2 --shard-index 0
# host B
uv run python scripts/ml_backfill_infer.py --manifest manifest.jsonl \
    --out r1.jsonl --shards 2 --shard-index 1
```

Re-running skips images already present in the output file. The inference host
needs onnxruntime with a GPU provider (CUDA, or ROCm for AMD) and read access to
the image store (e.g. the same NFS mount); it never connects to the database.

### Stage 3 ŌĆö ingest (needs the DB)

```bash
uv run python scripts/ml_backfill_ingest.py results.jsonl r0.jsonl r1.jsonl
```

Feeds predictions through the same DB pipeline as live inference (map ŌåÆ resolve
aliases/hierarchy ŌåÆ drop redundant ŌåÆ insert pending rows). Resumable via a
checkpoint file (`--checkpoint`, default `ml_backfill_ingest.done`); a row whose
image was deleted since the manifest was built is logged and skipped.

### Running Stage 2 on a separate AMD GPU host (ROCm)

`scripts/ml_gpu_infer.py` is an **app-free** twin of `ml_backfill_infer.py`
(only onnxruntime + numpy + pillow) so Stage 2 can run on a GPU box that can't
host the main app's stack. Its preprocessing/selection are copied from the
canonical code and drift-guarded by `tests/integration/test_ml_gpu_infer.py`.

Setup notes from an AMD Radeon RX 6800 XT (RDNA2 / gfx1030) on Linux Mint 22.3
(Ubuntu 24.04 "noble" base):

- The card's official consumer ROCm support ended at the 6.x line, so use
  **ROCm 6.4.4** (the last 6.x), not 7.x.
- The amdgpu kernel driver is already in-box (kernel 6.x exposes `/dev/kfd`); a
  **runtime-only** install is enough ŌĆö no DKMS:
  ```bash
  sudo usermod -aG render,video $USER   # then reboot
  wget https://repo.radeon.com/amdgpu-install/6.4.4/ubuntu/noble/amdgpu-install_6.4.60404-1_all.deb
  sudo apt install ./amdgpu-install_6.4.60404-1_all.deb && sudo apt update
  sudo apt install --no-install-recommends \
      rocm-core hip-runtime-amd rocblas miopen-hip hipblas hipfft rccl rocminfo rocm-smi-lib
  sudo ln -s /opt/rocm-6.4.4 /opt/rocm
  echo /opt/rocm/lib | sudo tee /etc/ld.so.conf.d/rocm.conf && sudo ldconfig
  ```
  (`hipblas`, `hipfft`, and `rccl` are load-time `NEEDED` deps of the onnxruntime
  ROCm EP even though single-GPU inference doesn't use them all.)
- onnxruntime-rocm ships **cp312** wheels (not 3.14), so use a Python 3.12 venv:
  ```bash
  python3.12 -m venv ~/ml-rocm && source ~/ml-rocm/bin/activate
  pip install numpy pillow \
      https://repo.radeon.com/rocm/manylinux/rocm-rel-6.4.4/onnxruntime_rocm-1.21.0-cp312-cp312-manylinux_2_27_x86_64.manylinux_2_28_x86_64.whl
  ```
- **`HSA_OVERRIDE_GFX_VERSION=10.3.0` is required** for gfx1030 ŌĆö without it HIP
  fails to init the kernels ("Module not initialized"). Verify the EP actually
  loads (not just that `get_available_providers()` lists it) by checking the
  runner prints `providers: ['ROCMExecutionProvider', ...]`.

```bash
export HSA_OVERRIDE_GFX_VERSION=10.3.0
python ml_gpu_infer.py --manifest manifest.jsonl --out results.jsonl \
    --model-dir ~/ml-models/swinv2_base_window8_256.dbv4-full \
    --storage-path /mnt/shuushuu --variant thumbs
```

**Throughput:** ~28 img/s on a 6800 XT at batch size 1 (~11 h for ~1.1M images
ŌĆö an overnight run, vs ~a week on the CPU box). It is GPU-bound at batch-1, so
running parallel shards does *not* increase aggregate throughput; batched
inference (the model has a dynamic batch dim) is the lever to go faster.

## docker-compose mounts

The `ml_models/` directory is mounted read-only into both services:

```yaml
# api and arq-worker services both include:
- ./ml_models:/app/ml_models:ro
```

The `*.onnx` file is gitignored ŌĆö only `README.md` and `selected_tags.csv` are
tracked.
